# Day 7 — Session Memory & Context in Multi-Agent Systems

This day covers how AI agents remember conversations across turns and how to scope data per user using context. Four files are included, each demonstrating a different piece of the memory puzzle.

---

## What Problem Are We Solving?

By default, every time you call `Runner.run()`, the agent starts with a blank slate. It has no memory of what you said before. This is fine for one-shot queries but breaks completely for any real chat application — the user would have to re-explain themselves every single message.

The solution is **sessions** — persistent stores that save conversation history and feed it back into each new turn automatically. This day covers three different storage backends for sessions, plus a separate concept called **context** which is about scoping live in-memory data per user during a single run.

---

## File 1 — `sessions_sqlite.py` (SQLite Session)

SQLite is a file-based database — no server, no setup, just a `.db` file sitting next to your code. This makes it the easiest way to get persistent memory working locally.

```python
session = SQLiteSession("demo_user", db_path="sessions.db")
```

The `session_id` is the key that separates different users' histories. Every message for `"demo_user"` is stored and retrieved from `sessions.db` automatically.

The agent is given the session at run time:

```python
result = await Runner.run(
    agent,
    input=user_input,
    session=session,
    run_config=RunConfig(
        session_settings=SessionSettings(limit=20),
        session_input_callback=keep_recent_history,
    ),
)
```

Two important controls are added here. `SessionSettings(limit=20)` caps how many messages are loaded from history — without this, a long conversation would eventually exceed the model's context window. The `session_input_callback` is a function that runs before each turn to trim history further:

```python
def keep_recent_history(history, new_input):
    MAX_HISTORY = 7
    return history[-MAX_HISTORY:] + new_input
```

This keeps only the last 7 exchanges, which is usually enough for conversational continuity without wasting tokens.

The CLI supports two special commands: `clear` wipes the stored session, and `exit` ends the program.

When to use SQLite: local development, single-server deployments, hobby projects, or any situation where you want zero infrastructure overhead.

The uploaded `sessions.db` file is the actual SQLite database generated by running this script.

## File 2 — `sessions_redis.py` (Redis Session)

Redis is an in-memory data store designed for speed. It is the standard choice for session storage in production web applications because it is fast, supports expiration of keys, and works across multiple server instances.

```python
session = RedisSession.from_url(
    session_id="demo_user",
    url="redis://127.0.0.1:6379/0",
)
```

The connection URL points to a local Redis instance on the default port. In production, this would be a hosted Redis service like Upstash, Redis Cloud, or AWS ElastiCache.

The rest of the code is nearly identical to the SQLite version — same history trimming callback, same `clear` command, same CLI loop. The only difference is where the data lives. SQLite writes to disk on the same machine. Redis writes to an in-memory store that can be shared across multiple machines, making it suitable for horizontally scaled deployments.

When to use Redis: production deployments, applications with multiple server instances, situations where you need session expiration (TTL), or high-traffic chat systems where database write latency matters.

## File 3 — `session_posql.py` (PostgreSQL Session via SQLAlchemy)

PostgreSQL is a full relational database. This is the right choice when your application already uses Postgres and you want session history stored alongside your other application data — users, orders, logs, etc.

```python
DATABASE_URL = "postgresql+asyncpg://postgres:password@localhost:5432/LOCAL"
engine = create_async_engine(DATABASE_URL, echo=False)
```

The `asyncpg` driver is used because the entire application is async. `create_async_engine` sets up the connection pool.

The session is created differently from SQLite and Redis:

```python
session = SQLAlchemySession(
    session_id="user-456",
    engine=engine,
    create_tables=True
)
```

`create_tables=True` tells the SDK to automatically create the required session history table in your database if it does not exist yet. This is convenient during development but you may want to manage migrations yourself in production.

This file does not add the history trimming callback — it trusts the database to hold the full history. This is fine for Postgres since you are not memory-constrained the way you might be with Redis, but for very long conversations you would still want to limit what gets sent to the model.

The engine is also explicitly disposed of after the chat ends:

```python
await engine.dispose()
```

This cleanly closes all database connections rather than letting them leak.

When to use PostgreSQL: production applications with existing Postgres infrastructure, multi-tenant systems, situations where you need to query or analyze conversation history, or compliance scenarios where you need auditable message logs.

## File 4 — `context.py` (RunContext — Per-User Scoped Data)

This file demonstrates something fundamentally different from the session files. **Context is not about persisting memory across runs — it is about scoping live data to a specific user within a single run.**

The use case here is a hotel booking system. Two users can interact with the same agent simultaneously, and their bookings must never mix.

The context is defined as a dataclass:

```python
@dataclass
class UserContext:
    user_id: str
```

An in-memory dictionary acts as the "database":

```python
db: Dict[str, List[str]] = {}
```

Tools access the current user's ID through `RunContextWrapper`:

```python
@function_tool
def book_room(wrapper: RunContextWrapper[UserContext], room: str) -> str:
    user_id = wrapper.context.user_id
    db.setdefault(user_id, []).append(room)
    return f"User {user_id} booked room {room}"
```

The same tool function handles any user correctly because it always reads the user ID from the context, not from a global variable. This is the key insight: the tool code is user-agnostic; the context makes it user-specific.

The agent is typed with the context:

```python
agent = Agent[UserContext](...)
```

And the context is passed per call:

```python
user_a = UserContext(user_id="A")
user_b = UserContext(user_id="B")

await Runner.run(agent, "book 202", context=user_b)
await Runner.run(agent, "view bookings", context=user_a)
```

User B books room 202. User A views their own bookings — which are empty, because the tool correctly scoped the operation to user A only.

When to use context: any time you have multiple users in the same system and need tool behavior scoped to whoever is making the request. This applies to bookings, shopping carts, personalized recommendations, or any stateful action that must not leak between users.

---

## Sessions vs Context — The Key Distinction

Sessions answer the question: "What did this user say before?"
Context answers the question: "Who is making this request right now, and what data belongs to them?"

They are not mutually exclusive. A real application would typically use both: a session to maintain conversational history across messages, and a context to pass the current user's identity and live state into tool calls.

Sessions are about time (past conversations). Context is about identity (current user scope).

---

## Storage Backend Comparison

SQLite is the simplest. One file, no dependencies, works immediately. It cannot be shared across multiple machines and has limited concurrency under heavy load. Use it to build and test locally.

Redis is the fastest. Keeps data in memory, supports automatic expiration, scales horizontally. Requires a running Redis server. Use it in production when speed and scalability matter.

PostgreSQL is the most powerful. Stores data in a structured relational format, supports complex queries, integrates with the rest of your application data. Requires more setup. Use it when you want session history as a first-class part of your data model.

## How to Run

Install dependencies:

```bash
pip install openai-agents python-dotenv sqlalchemy asyncpg redis
```

Run the SQLite chat (easiest, no extra services needed):

```bash
python sessions_sqlite.py
```

Run the Redis chat (requires Redis running locally):

```bash
redis-server  # in a separate terminal
python sessions_redis.py
```

Run the PostgreSQL chat (requires Postgres running locally with correct credentials):

```bash
python session_posql.py
```

Run the context demo (runs automatically, no input needed):

```bash
python context.py
```

---

## Summary

Day 7 is about giving agents memory and identity awareness. Without sessions, every conversation starts over. Without context, tools cannot know which user they are serving. Together, these two concepts are the foundation of any real-world multi-user AI application. The three session backends — SQLite, Redis, and PostgreSQL — represent a natural progression from local prototype to production deployment.
